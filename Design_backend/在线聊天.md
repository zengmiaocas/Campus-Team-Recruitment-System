# 一、开发目标

### 实现稳定的私聊与群聊功能，支持：

- 一对一私聊、项目群聊的消息发送与接收。
- 消息持久化存储与安全清空（仅清空自身记录）。
- 会话列表的自动聚合与未读消息数量计算。
- 群聊成员管理（队长移出成员，并触发系统通知）。
- 完善系统社交交互闭环，为用户提供基本的即时通讯能力。

# 二、核心任务

| 任务项 | 描述 |
|---|---|
| 聊天功能实现 | 支持私聊、群聊的消息发送、接收、存储与清空（仅自己不可见） |
| 会话列表管理 | 获取用户所有私聊/群聊会话，动态计算每个会话的未读消息数 |
| 消息记录管理 | 拉取历史消息时自动标记已读，支持清空后过滤已清除的消息 |
| 群聊成员管理 | 队长可移出群成员，被移出者状态变更，群内发送系统通知 |

# 三、核心接口实现

### 3.1 获取会话列表 GET /api/chat_list

**功能：** 返回当前用户所有有消息交互的私聊对象以及参与的群聊会话，并为每个会话计算未读消息数。

**请求：** 无参数（依赖 Cookie 中的 session_token）。

**实现逻辑：**

- 通过 session_token 获取当前用户手机号。
- 群聊列表：
  - 用户作为队长的所有未删除项目。
  - 用户作为队员且 applications.status = '已同意' 的所有项目。
- 私聊列表：从 messages 表中 DISTINCT 与当前用户有过消息往来的其他用户（排除自己）。
- 未读数计算：
  - 查询 chat_state 获取 last_read_msg_id 与 cleared_up_to_msg_id。
  - 根据 chat_type 和 target_id 统计 messages 表中 id 大于两者且 sender_phone != 当前用户的记录数（群聊还需排除系统消息？目前未排除，但系统消息应计入未读，合理）。
- 返回 JSON 列表。

### 3.2 获取聊天记录 GET /api/messages

**功能：** 返回指定会话的历史消息，并自动更新该会话的已读位置。

**参数（Query）：**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| type | string | 是 | private 或 group |
| id | string | 是 | 私聊时为对方手机号；群聊时为项目 ID |

**响应：** 消息数组，按 created_at 升序。

**实现逻辑：**

- 校验登录态。
- 获取当前用户在该会话的 cleared_up_to_msg_id（清空标记）。
- 根据 type 构建查询：
  - 群聊：chat_type='group' AND target_id = ? AND id > cleared_id。
  - 私聊：chat_type='private' AND ((sender_phone = me AND target_id = you) OR (sender_phone = you AND target_id = me)) AND id > cleared_id。
- 返回结果前对 content 和 sender_name 进行 HTML 转义。
- 若返回消息不为空，取最大 id 更新或插入 chat_state.last_read_msg_id。

### 3.3 发送消息 POST /api/send_message

**功能：** 向私聊或群聊发送一条消息。

**参数（表单）：**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| type | string | 是 | private 或 group |
| id | string | 是 | 私聊时为目标手机号；群聊时为项目 ID |
| content | string | 是 | 消息文本 |

**返回：** `{"msg": "发送成功"}`

**实现逻辑：**

- 获取当前用户。
- 插入 messages 表：(sender_phone = 当前用户, chat_type = type, target_id = id, content = content)。
- 不校验接收方权限（群聊入口由前端通过成员列表控制；私聊允许发给任何已注册用户）。

### 3.4 清空聊天记录 POST /api/clear_chat

**功能：** 清空当前用户在某会话中的聊天记录（仅自己视角，不影响他人）。

**参数（表单）：**

| 参数 | 类型 | 说明 |
|---|---|---|
| type | string | private 或 group |
| id | string | 会话标识 |

**实现逻辑：**

- 获取当前用户。
- 查询该会话中最大的消息 id（MAX(id)）。
- 更新 chat_state：
  - 若记录存在：UPDATE SET cleared_up_to_msg_id = max_id, last_read_msg_id = max_id。
  - 若不存在：INSERT 新记录，设置 cleared_up_to_msg_id = max_id, last_read_msg_id = max_id。
- 用户再次拉取消息时，由于 id > cleared_up_to_msg_id 条件，将看不到任何旧消息。

### 3.5 群聊成员管理 POST /api/kick_member

**功能：** 队长将某成员移出项目群聊，并发送系统通知。

**参数（表单）：**

| 参数 | 类型 | 说明 |
|---|---|---|
| proj_id | int | 项目 ID |
| target_phone | string | 被移出成员的手机号 |

**权限：** 当前用户必须是该项目的队长。

**实现逻辑：**

- 校验队长身份。
- 更新 applications 表中对应记录：status = '已移出', applicant_read = 0。
- 插入一条系统消息到群聊：

```sql
INSERT INTO messages (sender_phone, chat_type, target_id, content)
VALUES ('system', 'group', proj_id, '队长已将 【用户名】 移出队伍');
```

- 返回 `{"msg": "成功移出该成员"}`。

**后续影响：**

- 被移出的成员不再出现在群聊成员列表中。
- 他仍可查看旧消息（如已清空则看不到），但无法再接收新消息（因为轮询接口会过滤掉他的群聊权限）。
