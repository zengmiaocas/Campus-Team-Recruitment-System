# 一、开发目标

实现准实时的消息推送机制，让用户能及时收到新消息提醒，提升交互体验。

具体要求：

- 前端每隔 2.5 秒轮询后端，获取增量消息。
- 根据当前页面是否在对应聊天窗口，决定是否弹出 Toast 通知。
- 会话列表与聊天页面上展示未读消息红点，提醒用户及时查看。

---

# 二、核心任务

| 任务项 | 描述 |
|---|---|
| 增量消息轮询 | 设计 /api/poll_new 接口，根据客户端提供的 since_id 返回所有相关会话的新消息 |
| 弹窗通知 | 当收到新消息且当前不在该会话窗口时，在界面右下角显示 Toast 通知 |
| 消息红点提醒 | 全局消息图标显示红点（若有任一会话未读数 > 0）；会话列表中每个条目显示未读数 |
| 轮询状态维护 | 客户端持久化存储已读最大消息 ID（localStorage），并周期性更新 |

---

# 三、核心接口实现

### 3.1 增量轮询接口 GET /api/poll_new

**功能：** 返回当前用户所有相关会话（私聊 + 有权限的群聊）中，`id > since_id` 的新消息。

**参数（Query）：**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| since_id | int | 否 | 客户端已接收的最大消息 ID，默认为 0 |

**响应：** JSON 数组，按 `id` 升序排列。每条消息包含 `id`, `sender_phone`, `sender_name`, `chat_type`, `target_id`, `content`, `created_at`。

**实现逻辑：**

1. 获取当前用户手机号 `my_phone`。
2. 查询用户有权限的群聊 ID 列表 `my_groups`：
   - 作为队长的项目（`projects.leader_phone = my_phone`）
   - 作为队员且申请状态为"已同意"的项目（`applications` 关联）
3. 构建 SQL 查询新消息：

```sql
SELECT m.*, u.name as sender_name
FROM messages m
LEFT JOIN users u ON m.sender_phone = u.phone
WHERE m.id > ?
  AND m.sender_phone != ?                -- 排除自己发的消息
  AND (
    (m.chat_type = 'private' AND m.target_id = ?)          -- 发给我的私聊
    OR
    (m.chat_type = 'group' AND m.target_id IN (group_list)) -- 我所在的群聊消息
  )
ORDER BY m.id ASC
```

> **注意**：私聊条件 `m.target_id = ?` 只能收到别人发给自己的消息，自己是发送者时不会通过此接口重复通知（因为发送后前端已立即显示）。

4. 返回查询结果。

**性能说明：**

- 由于需要扫描 messages 表并按 id 过滤，建议为 `(id)` 和 `(chat_type, target_id, id)` 创建索引。
- 群聊列表 `IN` 子句中的项目数通常较少（最多几十个），可接受。

### 3.2 前端轮询策略（配合通知）

核心代码逻辑：

```javascript
let lastMsgId = localStorage.getItem(`lastMsgId_${myPhone}`) || 0;

setInterval(() => {
  fetch(`/api/poll_new?since_id=${lastMsgId}`)
    .then(res => res.json())
    .then(msgs => {
      if (msgs.length === 0) return;
      let maxId = lastMsgId;
      msgs.forEach(msg => {
        if (msg.id > maxId) maxId = msg.id;
        // 判断当前是否正在该聊天窗口
        let isInSameChat = (当前路径 === '/chat' &&
                            URL参数中type === msg.chat_type &&
                            URL参数中id === (msg.chat_type === 'group' ? msg.target_id : msg.sender_phone));
        if (!isInSameChat) {
          showToast(msg.sender_name || '系统', msg.content);   // 弹窗通知
          // 显示全局消息红点
          document.getElementById('msg-badge').classList.remove('hidden');
        }
      });
      lastMsgId = maxId;
      localStorage.setItem(`lastMsgId_${myPhone}`, lastMsgId);
      // 刷新当前聊天界面（如果正在聊天页面）
      if (当前在聊天页面) loadMessages();
      // 刷新会话列表（更新未读数）
      loadChatList();
    });
}, 2500);
```

**说明：**

- **弹窗通知**：仅在用户没有打开对应聊天窗口时弹出，避免干扰。
- **全局红点**：由 `loadChatList()` 根据每个会话的未读数总和决定是否显示；轮询过程中如果收到新消息且不在该会话中，会触发红点显示。
- **会话列表未读数**：每次轮询后都重新调用 `loadChatList`，该接口会重新计算每个会话的未读数（基于 `chat_state.last_read_msg_id` 与最新消息 ID 的差值），从而实现红点数字更新。

### 3.3 未读消息红点机制

- **全局红点**（页面右上角消息图标）：由 `chat_list` 接口返回的各个会话 `unread` 字段决定：若任一 `unread > 0`，则显示红点，否则隐藏。轮询收到新消息后，前端强制刷新 `chat_list`，从而更新红点状态。
- **会话列表红点**：每个会话条目后显示数字或小红点，直接使用接口返回的 `unread` 值。当用户打开某个会话，前端调用 `/api/messages` 拉取消息时，后端会更新该会话的 `last_read_msg_id`，后续轮询时该会话的 `unread` 将变为 0，红点消失。
- **聊天页面内红点**：聊天界面的侧边会话列表中同样展示未读数（复用 `chat_list` 接口），保持数据一致性。

### 3.4 登录态同步与心跳

为保证轮询时 session 仍然有效，前端需要定期检查登录状态：

```javascript
setInterval(() => {
  fetch('/api/check_session').then(res => res.json()).then(data => {
    if (data.status === 'logged_out') {
      // 清除本地数据并跳转到登录页
      localStorage.removeItem(`lastMsgId_${myPhone}`);
      alert('登录已过期，请重新登录');
      window.location.href = '/login';
    }
  });
}, 3000); // 也可与轮询合并
```
